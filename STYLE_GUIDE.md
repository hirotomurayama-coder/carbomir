# Carbomir 執筆スタイルガイド

Carbomir のコンテンツ品質を一定に保つための執筆規範。AI ドラフト生成 (`npm run ai:draft`) と人手執筆の両方が遵守する。

---

## 0. 大方針

- **主ペルソナ**: 事業会社の CSR / サスティナビリティ担当 (中〜上級レベルの知識)
- **主用途**: 規制変更のキャッチアップ。**取締役会や社内向け説明資料への転用**
- **主差別化**: 「編集部の論点」セクション。生情報はコモディティ、解釈で差別化する
- **執筆の構え**: 確信のある事実は断定、解釈は中立的、予測は明示的に弱く

---

## 1. 確信度 3 段階の使い分け

| 確信度 | 用法 | 文末表現の例 |
|---|---|---|
| **強 (断定)** | 一次資料で確認できる事実、制度規定 | 「~ **である**」「~ **する**」「~ **となる**」「~ **が定められている**」 |
| **中 (解釈)** | 編集部の解釈・整理 | 「~ **と考えられる**」「~ **と整理できる**」「~ **という見方が支配的**」 |
| **弱 (予測)** | 未確定領域、未来予測 | 「~ **の可能性がある**」「~ **も論点となり得る**」「~ **の方向で検討されている**」 |

### NG パターン (現状の seed に多い曖昧表現)
- ✗ 「~ という側面が強い」 → 解釈なら「~ と考えられる」、事実なら断定
- ✗ 「~ という意味合い」 → 解釈なら「~ と整理できる」
- ✗ 「~ と言われている」 → 誰が言っているかが書ければ断定、書けなければ削除

## 2. 出典の示し方 (ハイブリッド)

### 末尾集約 (Sources セクション)
- ページ末尾の `Sources` セクションに `{label, url}` の形で集約
- 一般的な背景解説の出典はここに

### インライン引用 (本文中)
以下は **必ず本文中にインライン引用** する:
- **規制の発効日・条件** — 例: 「2026 年 4 月から ([経産省 GX 政策](https://...)) ~」
- **価格・統計数字** — 例: 「平均価格は 50 USD/t ([CarbonPlan OffsetsDB](https://...) 集計, 2026-05)」
- **企業の公式声明** — 例: 「Microsoft は ~ と公表 ([2024 Sustainability Report](https://...))」

### Markdown 記法
```markdown
... 2026 年 4 月から ([経産省 GX 政策](https://www.meti.go.jp/...)) 有償化される。
```

## 4. 用語・表記統一

### 略号・固有名詞
| 用語 | 規範 | 例 |
|---|---|---|
| Verra | 初出は「Verra (VCS)」、以降は文脈で使い分け | Verra (機関) / VCS (スキーム) |
| GX-ETS | **ハイフン保持** | GX-ETS (○) / GX ETS (✗) / GXETS (✗) |
| J-クレジット | **ハイフン保持** | J-クレジット (○) / J クレジット (✗) |
| JCM | 大文字 | JCM (Joint Crediting Mechanism) |
| SBTi | そのまま | SBTi、Science Based Targets initiative |
| TCFD / CDP | 大文字 | TCFD, CDP |
| ICVCM | 大文字 | ICVCM |

### 数値・日付・通貨
| 項目 | 規範 | 例 |
|---|---|---|
| 西暦 | **半角数字 + 半角スペース + 「年」** | 「2026 年」(○) / 「2026年」(✗) |
| 月日 | 「2026 年 4 月」「2026-05-21」(ISO) | ISO は技術文脈、和暦表記は本文中 |
| 通貨 | USD/t を基本 | 「50 USD/t」 (○) / 「$50/t」「ドル/t」(✗) |
| 数量 | 千の位カンマ + 単位 | 「11,000 件」(○) / 「11000件」(✗) |
| トン CO2 | 「t-CO2」or「tCO2e」(初出時に明示) | 「年間 100 万 t-CO2」 |

### カタカナ語の使用
- 業界用語はカタカナ優先: メソドロジー / スタンダード / クレジット / レジストリ / プロジェクト / オフセット
- 「方法論」「規格」等の和訳は使わない (専門性低下を避ける)

### 「ボランタリー」「コンプライアンス」
- 「ボランタリー市場」/「ボランタリークレジット」
- 「コンプライアンス市場」/「遵守クレジット」
- 「自主市場」「強制市場」等の表記は使わない

## 5. 見出し階層

### Entity / Case Study / Timeline event の本文 (Markdown)
- Section の `body` 内では `## ` (H2) で開始
- サブセクションは `### ` (H3)
- H1 はページタイトルに使われるので body 内では使わない

### 強調
- `**太字**` を **構造化** に使う (重要キーワード、判断軸)
- `*斜体*` は控えめに (引用、強調)
- 段落の連続太字は避ける

### リスト
- 箇条書きを積極使用 (実務者は走査読みする)
- ネストは 2 階層まで
- 番号付きリストは順序が意味を持つときのみ

---

## 6. 編集論点セクション

### セクション名は統一: **「編集部の論点」**
- 過去の seed には「Carbomir の見方」「編集部の見解」等が混在 → **統一**
- 「日本企業が参考にできる点」のような他の定型句は **使う場面を限定**:
  - ケーススタディ末尾: OK
  - Entity / Timeline: 不要 (代わりに「編集部の論点」に集約)

### 編集論点に書くこと
1. **規制 / 制度がなぜそうなったか** — 政策的背景の解釈
2. **何が変わるか** — 既存からの変更点
3. **誰が影響を受けるか** — affected entities / industries
4. **日本企業にとっての含意** — どう備えるべきか
5. **未確定点と論点** — 議論余地のあるトピック (確信度弱)

### 長さ
- Entity / FAQ: 1-2 段落 (200-400 字)
- Timeline event: 2-3 段落 (300-600 字)
- Case study: 3-5 段落 (400-800 字)

---

## 7. 「要確認」マークの扱い

### 公開コンテンツには残さない
- 「(要確認: ...)」「(要確認)」は **公開前 (status: draft) で必ず解消**
- 解消方法:
  - a) 一次資料で確認 → 該当箇所を断定に書き換え
  - b) 確認困難 → 該当箇所を削除 or 範囲縮小
  - c) 構造的に不確実 → 「(運用注視: ...)」ラベルに置換

### 「運用注視」ラベルとは
- 制度が動き始めていない、運用実態がまだ見えない等、**本質的に不確実な情報**
- 例: 「(運用注視: GX-ETS 第2フェーズの細則は 2026-Q4 に公布予定)」
- 透明性として公開時にも残す

### 内部管理
- `/editorial` ダッシュボードで「公開ページに残る要確認」を集計 (アラート的に)
- AI ドラフト生成時はまだ「(要確認: ...)」を出すが、承認時に編集者が解消

## 8. 関連エンティティ・リンクの貼り方

### 内部リンク
- `[GX-ETS](/entities/gx-ets)` のように Markdown リンクで明示
- 初出時のみリンク化、以降は素の表記でよい
- ただし「編集部の論点」内では論点ごとに該当エンティティへ再リンクしても OK

### 関連エンティティの宣言
- `related_entity_slugs: ["gx-ets", "jcredit"]` でデータモデル上の関連を宣言
- UI 側で自動的にサイドカードに表示される

### 関連比較行列
- `related_matrix_slugs: ["credit-eligibility"]` で比較行列への導線を貼る

## 9. データモデルへの値の入れ方

### `last_reviewed_at`
- レビュー日 (執筆日 or 修正日) を ISO 形式で
- 例: `"2026-05-21"`

### `next_review_at` (Phase Γ で追加予定)
- 次回レビュー予定日
- 規制関連: 1-3 ヶ月後
- 一般エンティティ: 3-6 ヶ月後

### `status`
- `draft` — 執筆中、AI ドラフト承認待ち
- `published` — 公開中 (これだけが UI に出る)
- `archived` — 過去版、参照用

### `tags`
- `src/lib/types.ts` の `TAG_VOCABULARY` から選ぶ
- 1 エンティティに 3-6 タグが目安

### `policy_status` (regulation entity 用)
- `proposed` — 提案段階
- `enacted` — 法制化済み
- `effective` — 施行中
- `under_review` — 改訂審議中

### `next_milestone` (regulation entity 用)
- 次のマイルストーン日付 + 内容
- 例: `"2027-04-01: 第2フェーズ有償化開始"`

## 10. AI ドラフト生成での適用

`npm run ai:draft` の system prompt はこのスタイルガイドの主要ルールを反映する:
- 確信度 3 段階
- 「編集部の論点」セクション名
- 「(要確認: ...)」を生成段階では許可 (承認時に解消)
- 用語規範
- 内部リンク + 関連エンティティ宣言

具体的なプロンプトの実装は `scripts/ai-draft.ts` の `COMMON_RULES` 定数を参照。
